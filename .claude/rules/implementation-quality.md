# implementation-quality.md — 実装品質ルール

エージェントは実装作業前に本ファイルを読み、違反しないこと。

## 禁止事項

1. **中身のないスタブを「完了」と報告しない**
   - `return null` / `// TODO` / `throw new Error("not implemented")` のままタスクを `cc:DONE` にしない。
   - 部分実装の場合は `cc:WIP` のまま残し、何が残っているかを Plans.md に明記する。

2. **「動くフリ」を作らない**
   - UI だけ動いて API は固定モックを返す、というのは MVP として明示的にユーザー合意がある場合のみ許可。勝手にやらない。

3. **不要な抽象化を入れない**
   - 「将来こうなるかも」で interface / 拡張ポイントを作らない。3 回同じパターンが現れてからリファクタする。

4. **不要なエラーハンドリングを入れない**
   - 「起こり得ないケース」への try/catch は書かない。境界（ユーザー入力・外部 API）のみ。

5. **ユーザーが指示しないリファクタを混ぜない**
   - バグ修正タスクのついでに無関係なクリーンアップをしない。差分は最小限に。

## 必須事項

1. **既存ファイル優先**
   - 新規ファイルを作る前に、既存ファイルへの追記で済まないか確認する。

2. **コメントは「Why」のみ**
   - 「何をするか」は識別子で表現する。「なぜそうするか」（非自明な制約・回避策）のみコメント。

3. **不明点は実装前に質問する**
   - 要件定義.txt の「予想で補わず必ずヒアリングを」を遵守。
