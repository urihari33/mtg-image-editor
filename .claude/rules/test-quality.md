# test-quality.md — テスト品質ルール

エージェントはテスト作成・修正の前に本ファイルを読み、違反しないこと。

## 禁止事項

1. **テストを通すためにテストを書き換えない**
   - 実装が壊れていてテストが落ちた場合、テスト側を緩めて通すのは禁止。実装を直す。
   - `expect(x).toBe(y)` を「とりあえず `toBeTruthy()` に変える」のような書き換えは絶対にしない。

2. **アサーションのない「実行できるだけ」のテストを書かない**
   - `expect` がない / `assert` がない テストはレビューで落とす。

3. **モックでプロダクション挙動を書き換えない**
   - 「Scryfall API が遅いから常に空配列を返すモックにする」のような、現実と乖離するモックは禁止。
   - 統合テストは可能な限り実 API（または記録したフィクスチャ）を使う。

4. **`it.skip` / `test.skip` を残したまま `cc:DONE` にしない**
   - スキップする場合は理由をコメント＋ Plans.md にフォローアップタスクを追加。

## 必須事項

1. **失敗を再現するテストを先に書いて、それから実装を直す**（バグ修正時）
2. **テストの命名は「振る舞い」で書く**: `"検索ボックスに日本語を入力するとScryfallのlang=jaが呼ばれる"` のように、入力と期待出力が読めるように。
3. **副作用テストは isolation**: localStorage / fetch のモックはテスト後にリセット。
